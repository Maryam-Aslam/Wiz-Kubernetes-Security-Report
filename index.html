Wiz Kubernetes Security Report

Limited 2025 Refresh and Key Takeaways

The 2025 Kubernetes Security Report is based on our analysis of over 200,000 cloud accounts and continues to uncover prevalent risks in K8s environments that often go unnoticed.

In our previous Kubernetes Security Report, we provided statistics on the state of Kubernetes security. That report highlighted the increasing amount of threats targeting these environments and also uncovered unnoticed risks by attributing them to various stages of the attack path on Kubernetes clusters, from initial access to impact.

The main takeaway: Overall container security maturity is low, even though these environments are appealing targets for attackers and attempts to breach them are on the rise. The report also identified the weakest links in cloud environments and introduced "The Zone Defense" as an effective defense strategy for mitigating them.

Given that it's been more than 12 months since that last report, and Kubernetes remains a hugely popular technology, we decided to publish a limited refresh. Rather than going deep into multiple aspects of Kubernetes security, we've chosen a few key areas to focus on. We now have the luxury of multiple years' worth of data, so this report is made up of a subset of the most interesting stats from last year, as well as some shiny new data. In this report, we start with the a few key takeaways and continue with the detailed analysis of the statistics collected. Let's dive in.

Key Takeaways

Kubernetes Environments Continue to Be a Central Target

Malicious parties are as active as ever, going after Kubernetes API endpoints and data plane services.

We repeated our experiment from the previous year, tracking the amount of time between initially staging the new cluster and the first malicious probing attempts. As with last year, the attacker response times show that the newly created public clusters quickly become targets. This time the longest attacker response time for GKE clusters was 1 hour 15 minutes vs 2 hours 13 minutes last year. For EKS and AKS clusters this "grace period" is drastically shorter — 28 and 18 minutes respectively.

This is admittedly a startling figure, but readers shouldn't focus too much on the exact numbers here; the point is that in general, when staging the cluster, K8s operators must take care of the defenses from the start and not roll them out subsequently. This is especially true for protections against initial access. We discussed extensively various ways the attackers can get a foothold in your cluster in our blog, "Making Sense of Kubernetes Initial Access Vectors."

For the information regarding post-exploitation activities, we turned to our sensor data. Among other platforms, Wiz Sensor is deployed on Kubernetes worker nodes and is a source of valuable statistics on attackers' behavior post initial compromise. Two prominent cases for this are the Dero cryptojacking campaign with Pause containers and the SeleniumGrid threat campaign. In the first case, malicious actors abused the misconfiguration in K8s server API authentication to deploy a set of cryptomining workloads masquerading as legitimate control plane workloads. In the second case, attackers exploited the lack of authentication set by default in the Selenium WebDriver API to upload XMRig cryptominer on Selenium worker nodes. Further campaign analysis by Cato also confirmed the use of the compromised resources for proxyjacking.

Kubernetes SecOps are Maturing

Kubernetes users are more comfortable with operating clusters. Operating and security processes have matured. This is evident from the faster version upgrades, decreased vulnerabilities and better workload security posture.

Over the course of this research project, we paid special attention to the version analytics. Overall, version distributions show that maintenance hygiene has improved compared to the previous year. For example, the most popular version as of October 2024 is 1.29 (released 12/2023) versus 1.24 (released 5/2022) a year ago. Cloud Service Providers (CSPs) generally offer a similar, albeit delayed, support span. Hence, prompt version migration in K8s clusters is extremely important. At the moment of data collection (October 2024, a year after the original 2023 data sample), all versions prior and including 1.27 have actually passed upstream EOL, which constitutes 46% of total clusters (compared to 58% last year). Workload Security Posture Improvements

There are still ample opportunities for attackers to move laterally within the cluster — either through the usage of the compromised workload permissions or through a worker node as a shared medium. We see, however, a significant improvement in security posture of the workloads compared to last year.

Multiple metrics indicate an overall improvement in the security posture of data plane pods. Most Kubernetes pods run with service accounts that facilitate communication with the API server. If a pod is compromised, the attacker gains access to its associated service account, which is mounted as a file inside the pod. This makes proper RBAC permissions configuration critical. Year over year, we've observed a significant decline in the number of pods with high Kubernetes privileges, as defined by the Wiz permissions model:

There are two possible explanations for this trend: first, improved vulnerability management processes within organizations, including faster image patching and more strategic image selection. Second, an overall improvement in container image quality.

Adoption of Security Features is Lagging

About a year ago, AWS introduced EKS Access Management — a shiny new feature allowing a novel way for IAM users and IAM roles to connect to and manage EKS clusters and resources via EKS API. We have analyzed the security implications of this feature in a two-part blog series. Following this introduction, AWS deprecated the previous authentication method, CONFIG_MAP.

In last year’s report we talked extensively about the lack of adoption of security features, so we were naturally curious about the adoption of this new authentication mode. Numbers show that after over a year of being generally available, the API authentication mode is used in only 3% of clusters. The mixed authentication mode, API_AND_CONFIG_MAP, is employed in 16% of clusters, and an overwhelming majority—81% of clusters—still exclusively use the old CONFIG_MAP authentication mode.
